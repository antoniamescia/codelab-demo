What is an AI Agent?

An AI agent is a program that doesn't just answer in one shot — it works through a loop: it thinks about what needs to be done, uses a tool to act on the world, observes what that tool returned, and then decides the next step based on that result. You can see this exact loop in how this very file was created: I thought "I should check the project folder before writing anything," so I used a tool (a shell command to list the directory), observed that it was empty, and then decided the next step was to write this HTML file directly instead of editing something that didn't exist. That cycle — think, act, observe, decide — repeated as many times as needed, is what separates an agent from a simple chatbot that only replies with text.

Subagents

A subagent is essentially an instance of the agent that gets handed a scoped task and works with its own context window, separate from the main conversation and from any other subagent running at the same time. This means it doesn't carry the full history of the chat with the user, nor does it see what its "peers" are reasoning through: it only knows the instruction it was given and whatever it discovers on its own while working. That separation isn't a minor technical detail, it's precisely what makes running things in parallel possible: if two subagents shared the same context, their intermediate steps, mistakes, and decisions would blend together and pollute each other's reasoning. By keeping them isolated, the main agent can launch several subagents at once to explore code, research, or make changes simultaneously, then simply collect each one's final result, much like asking different colleagues to look into something independently and report back only the conclusion.

Git Worktrees

A git worktree is a second working directory linked to the same repository, checked out to its own branch with its own files sitting on disk. Instead of one folder where the checked-out branch and file contents change every time you switch context, you get several folders that all share the same underlying .git history but can hold different code at the same time. That is exactly what makes it safe to run multiple coding agents side by side: each agent gets its own worktree to read from and write to, so its uncommitted edits, half-finished refactors, and generated files never land in the same directory as another agent's. Only once an agent commits does its work rejoin the shared history, at which point the usual git merge rules take over instead of two processes silently overwriting the same lines on disk.

Orchestration Workflows

An orchestration workflow is a predefined script that decides, ahead of time, how work moves between agents: which subagent runs first, which steps can fan out to run in parallel, and where a verification pass has to pass before anything continues. The key difference from a single agent handling a list of requests in conversation is who is doing the deciding: in a plain chat, the model itself improvises the next step based on context, whereas in a workflow the sequence, branching, and hand-offs are fixed by code, so the same input reliably produces the same shape of process. This matters because improvisation is flexible but unpredictable, while a workflow trades some of that flexibility for repeatability — the same stages, checks, and outputs every time, even as the underlying model changes. In practice this looks like one agent drafting a plan, several subagents working pieces of it at once, and a dedicated step that checks the combined result before it's considered done.
